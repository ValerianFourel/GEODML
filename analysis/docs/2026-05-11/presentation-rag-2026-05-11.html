Why two of everything

slide 03 · design rationale

The five binary axes are not redundant — each strips a different alternative explanation off the table.

2 search engines  →  isolates SERP-distribution bias
                     (searxng = federated metasearch, ddg = independent index)

2 LLMs            →  isolates model-family idiosyncrasy
                     (Llama dense decoder  vs.  Qwen MoE-style training mix)

2 pool sizes      →  controls candidate-set effect
                     (top-20 = harder discrimination, top-50 = more tail)

2 prompts         →  manipulates instruction-induced bias
                     (biased "marketing assistant"  vs.  neutral "search system")

2 methods         →  manipulates available evidence
                     (snippet-only ≈ 150 chars  vs.  RAG ≈ 3 × 800-char chunks)

If a treatment effect survives all four non-treatment splits, it is not an artefact of one engine, one model, one prompt, or one candidate-set size. That is the robustness story the paper wants.

The two pool sizes

slide 06 · axis 3
Pool = how many candidate URLs the LLM must rank.

   top-20  ┌────────────────────┐
           │ 20 URLs per SERP   │  ← harder discrimination, denser bias signal
           └────────────────────┘

   top-50  ┌──────────────────────────────────────────────┐
           │ 50 URLs per SERP                              │  ← more tail,
           └──────────────────────────────────────────────┘    more noise

Both feed the same top-10 final cut for evaluation. The pool size controls how much diversity / tail the LLM sees before deciding.

A treatment that only shows up at one pool size hints at a "rare-candidate" effect; one that holds across both is robust.

The two methods — the new axis

slide 08 · axis 5
SNIPPET-ONLY (baseline)
1. [example.com] Title — snippet (~150 chars)

The LLM sees only what a normal SERP UI shows.

RAG (query-conditional, like production GEO)
1. [example.com] Title
   snippet : ...
   passage : <chunk #7  sim 0.71>
             --- <chunk #14 sim 0.65>
             --- <chunk #2  sim 0.61>

Top-3 chunks of that page's body closest to this keyword, recomputed per (kw, url).

This is the manipulation that matters for the GEO claim: production systems (ChatGPT search, Perplexity, Bing Copilot) all do query-conditional retrieval. Snippet-only is the academic baseline; RAG is the field reality.

How the RAG index was built

slide 09 · one-time, ~$2.72
44,220 unique URLs across 4 (engine, pool) cells
        │
        │   trafilatura.extract(html, max_chars=50_000)
        ▼
Full page bodies  ────────────────────────  ~136 M tokens total
        │
        │   recursive char chunker (size=800, overlap=200, min=100)
        ▼
906,322 chunks       (mean 24.8 chunks per page)
        │
        │   OpenAI text-embedding-3-small  (1536-dim, L2-normalised)
        ▼
chunk_embeddings.npy   per cell  (~1.3 GB × 4 = 5.3 GB on disk)

For each keyword (4,011 total):
        keyword_embeddings.npy   (1536-dim)

At rerank time, for each (keyword, url) on the SERP:
        sims    = chunk_emb[url] @ keyword_emb[kw]      # cosine
        passage = " --- ".join(top-3 chunks by similarity)
        →  cached in retrieved_top3.parquet  (~13 K rows / cell)

Sanity check: keyword "educational software for schools" → top global chunk on searxng_top20 came from research.com/software/best-education-software with cosine similarity 0.654. Semantically appropriate.

Results approach — DML on rank_delta

slide 11 · estimation

For every (treatment, cell, learner, …) we fit a Double Machine Learning estimator (econml.LinearDML or partial-linear) with:

  • Outcome Y = rank_delta = pre_rank − post_rank. Positive = LLM promoted, negative = LLM demoted.
  • Treatment T = one of 10 on-page features (e.g. T7_source_earned = 1 if URL is on a known earned-media domain).
  • Confounders X = ~20 covariates (domain, keyword cluster, position prior, query-intent class, length, …).
  • Nuisance learner = LightGBM (chosen by cross-fit comparison last week).

Headline slice: POOLED · plr · lgbm · rank_delta — pools across the four (engine, pool) cells for power, partial-linear with LightGBM, on rank-delta.

What the paper can claim

slide 17 · contributions
  1. Robustness. T7 earned-media demotion survives the move from snippet-only to query-conditional retrieval.
    −1.61*** → −1.27*** for biased; −0.42*** → −0.50*** for neutral.
  2. Anchoring on domain-quality signals. For T7 / T3 / T4a, RAG narrows the prompt-bias gap by 35–66 %. Real GEO systems partially neutralise prompt-instruction bias on this class of signals.
  3. Selective amplification on content-relevance signals. For T5, retrieval multiplies the biased prompt's competitor-preference by ≈ 9×. Production systems may strengthen prompt-driven preferences when retrieval supplies confirming evidence.
  4. Asymmetric interaction. RAG impacts the biased arm more strongly than the neutral arm in both directions — "retrieval changes behaviour more when the prompt is looking for something the retrieval can confirm or contradict."

Caveats

  1. N smaller in RAG. snippet POOLED ≈ 20 k obs/variant; RAG POOLED ≈ 12 k (pages without retrievable chunks drop). CIs widen.
  2. T5 has only one significant arm. Frame: "biased becomes sig. and 3.4× larger under RAG", not "RAG flips T5".
  3. order_probe is 6/32. Jaccard / OAK numbers land after resume.
  4. Qwen RAG untested in Phase 3 — could behave differently at the rank-level granularity.
  5. Naïve char chunker, not semantic. Sensitivity check is on the future-work list, not the critical path.
